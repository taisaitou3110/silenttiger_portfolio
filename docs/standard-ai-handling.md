# AI処理・エラーハンドリング標準仕様 (v1.0)

## 1. 概要
本プロジェクトにおけるAI（Gemini API等）との連携において、API制限（429 Too Many Requests）やサーバー混雑（503）を賢く回避し、ユーザーにストレスを与えないための標準的な実装方法を定義します。

### 基本理念
- **レジリエンス（回復力）**: 1つのモデルが制限されても、代替モデルで処理を完遂する。
- **透明性**: 内部の処理状況（思考中、リトライ中、完了）をリアルタイムに可視化する。
- **プラン非依存**: 契約プラン（無料/有料）が変わっても、設定値の変更のみで最適化できる構造にする。

---

## 2. 処理レベルに応じた3つの階層

データのサイズや重要度に応じて、以下の3段階のハンドリングを使い分けます。

| レベル | 対象処理の例 | 推奨戦略 |
| :--- | :--- | :--- |
| **Level 1: Standard** | 単発のテキスト生成、添削 | モデルフォールバック（身代わりリトライ） |
| **Level 2: Intensive** | 画像解析、長文要約 | モデルフォールバック + 指数バックオフ待機 |
| **Level 3: Distributed** | 大量資料の学習（RAG） | クライアント主導のバッチ逐次送信 + スロットリング |

---

## 3. サーバー側共通ロジック (`lib/ai-handler.ts` 想定)

### 3-1. モデル・フォールバック戦略
特定のモデルに依存せず、環境変数で定義された優先順位に従って「生きているモデル」を探します。

```typescript
// 実装イメージ
const MODEL_ORDER = (process.env.AI_MODEL_FALLBACK_LIST || "gemini-2.0-flash,gemini-flash-latest,gemini-pro-latest").split(',');

for (const modelName of MODEL_ORDER) {
  try {
    // 処理実行
    return await executeTask(modelName);
  } catch (error) {
    if (isRateLimit(error)) continue; // 次のモデルへ
    throw error;
  }
}
```

### 3-2. 指数バックオフ
429エラー検知時、即座にエラーを返さず、待機時間を倍増させながら再試行します。
- **計算式**: `WaitTime = Math.pow(2, retryCount) * BaseTime`

## 4. クライアント側（UI/UX）標準

### 4-1. AIProcessOverlay の統合
AI処理中は必ず `AIProcessOverlay` を表示し、以下のステータスを遷移させます。

1. **`transfer`**: データのアップロード、テキスト抽出中。
2. **`thinking`**: AIへのリクエスト送信、およびリトライ待機中。
3. **`generating`**: 回答の合成中。
4. **`completed`**: 処理完了。

### 4-2. 抽象化されたメッセージング
将来の高速化を見据え、エラーを直接見せずポジティブな表現に変換します。
- ❌ 「API制限中です。20秒待ってください」
- ✅ 「システムを最適化しています。まもなく再開します...」

## 5. 有料プラン移行への備え（スケーラビリティ）

### 5-1. 設定の外部注入
以下のパラメータはハードコードせず、環境変数または設定ファイルから読み込みます。

| パラメータ名 | 説明 | 無料枠の推奨値 | 有料枠の推奨値 |
| :--- | :--- | :--- | :--- |
| `AI_MODEL_FALLBACK_LIST` | 試行するモデルの優先順位 | `2.0-flash,flash-latest,pro-latest` | `1.5-pro` (最高精度一本勝負) |
| `AI_BATCH_SIZE` | 1回あたりのリクエスト件数 | 15 - 20 | 100+ |
| `AI_THROTTLE_MS` | リクエスト間の待機時間 | 2000 - 3000 | 0 - 100 |
| `AI_MAX_RETRIES` | 1モデルあたりの最大リトライ数 | 5 | 2 |

### 5-2. TaskType の明示
Embedding（ベクトル化）時は、用途に合わせて `taskType` を必ず指定します。
- 学習時: `RETRIEVAL_DOCUMENT`
- 検索時: `RETRIEVAL_QUERY`
これにより、API側のリソース消費が最適化され、制限にかかりにくくなります。
